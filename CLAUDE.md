# プロジェクトルール

## ドキュメント構成ルール

新機能・新プロダクトの開発を始める前に、以下のドキュメントを必ず作成・整備すること。

### 必須ドキュメント

| ファイル名 | 目的 | 主な記載内容 |
|---|---|---|
| `product.md` | AIにプロダクト理解させる | プロダクトの概要・目的・ターゲットユーザー・解決する課題 |
| `requirements.md` | 機能理解 | 機能要件・非機能要件・ユースケース・制約条件 |
| `architecture.md` | 実装方針 | 技術スタック・システム構成・設計方針・ディレクトリ構成 |
| `api.md` | コード生成の基準 | エンドポイント定義・リクエスト/レスポンス仕様・認証方式 |
| `schema.md` | DB生成の基準 | テーブル定義・ER図・インデックス・制約 |
| `tasks.md` | 実装順の管理 | タスク一覧・依存関係・優先順位・実装フェーズ |

### ドキュメントの配置

```
docs/
├── product.md
├── requirements.md
├── architecture.md
├── api.md
├── schema.md
└── tasks.md
```

### 追加ドキュメント

上記6ファイルに収まらない情報は、必要に応じて `docs/` 配下に追加ファイルを作成する。

例：
- `docs/auth.md` — 認証・認可の詳細設計
- `docs/deployment.md` — デプロイ手順・環境構成
- `docs/testing.md` — テスト方針・テストケース
- `docs/data-flow.md` — データフロー・バッチ処理の設計

### 運用ルール

- 実装変更に伴いドキュメントが古くなった場合は、**実装と同じタイミングで更新**する
- 各ドキュメントの先頭に最終更新日を記載する（例: `> 最終更新: 2026-03-06`）
- `tasks.md` は実装の進捗に合わせてステータスを更新する
